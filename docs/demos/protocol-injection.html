Blocking Protocol Injection
in MCP Proxies

An attacker sends SQL injection payloads and malformed JSON through the proxy. Without strict mode, these bypass every inspection stage and reach the upstream server.

The Attack

MCP uses JSON-RPC 2.0 as its wire format. But a proxy that only checks for valid JSON -- or does not check the body at all -- will forward any POST request to the upstream server.

An attacker can exploit this by sending non-MCP payloads through the proxy: SQL injection statements, shell commands, custom JSON structures, or even binary data. Because the proxy does not recognize these as MCP requests, it skips all session, policy, and behavior checks. The payload arrives at the upstream server un-inspected.

This class of vulnerability turns the MCP proxy into a protocol-level tunnel. The attacker does not need a valid session, a registered agent, or a matching policy. They simply bypass the entire security stack by not speaking MCP.

The Demo

Two attack variants: a plain text SQL injection payload, and a JSON body that is not valid JSON-RPC 2.0.

attack-1.sh SQL injection via plain text POST
# Plain text body, not JSON, not JSON-RPC curl -s -X POST http://127.0.0.1:8080 \ -H "Content-Type: text/plain" \ -d 'DELETE FROM users WHERE 1=1;'
attack-2.sh Arbitrary JSON (not JSON-RPC 2.0)
# Valid JSON but missing "jsonrpc": "2.0", not MCP curl -s -X POST http://127.0.0.1:8080 \ -H "Content-Type: application/json" \ -d '{"action": "drop_table", "target": "users"}'
response HTTP 403 (both attacks)
{ "error": "NON_MCP_REJECTED", "message": "Only valid MCP JSON-RPC 2.0 requests are accepted in strict mode" }

The Defense

Arbiter stops this attack at Stage 5 of its pipeline: MCP protocol parsing.

arbiter-mcp (Stage 5)

When strict_mcp = true (the default), the MCP parser validates every POST body against the JSON-RPC 2.0 specification. It checks for the required "jsonrpc": "2.0" field, valid method names, and well-formed params. Any POST body that fails this check is rejected with HTTP 403 and error code NON_MCP_REJECTED. The request never reaches session enforcement, policy evaluation, or the upstream server.

This closes the entire class of protocol bypass vulnerabilities. The proxy is no longer a tunnel -- it is an MCP-aware gateway that only forwards traffic it can fully inspect.

config.toml Strict MCP mode is on by default
[proxy] upstream_url = "http://127.0.0.1:8081" require_session = true strict_mcp = true # reject all non-JSON-RPC 2.0 POST bodies

Try It Yourself

terminal Run the demo locally
git clone https://github.com/cyrenei/arbiter-mcp-firewall cd arbiter && cargo build --release cd demos/02-protocol-injection bash demo.sh

The script runs both attack variants -- plain text and malformed JSON -- and shows Arbiter rejecting each one.

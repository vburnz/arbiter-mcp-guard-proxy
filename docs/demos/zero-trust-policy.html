Deny-by-Default:
Why AI Agents Need Zero Trust

A rogue contractor's agent passes session checks and has the tool in its whitelist. But the policy engine finds no matching Allow rule. Deny-by-default blocks the request with a full policy trace.

The Attack

Session whitelisting controls which tools an agent can call. But it does not control which agents can call which tools with which parameters for which principals. That is the job of the policy engine.

In this scenario, a single Allow policy permits deploy_service only when the principal is user:trusted-team and the intent contains the keyword "deploy". An attacker registers an agent owned by user:rogue-contractor and creates a session with deploy_service in the whitelist.

The session check passes -- the tool is in the whitelist. But the policy engine evaluates all policies and finds no match. The principal criterion requires user:trusted-team, not user:rogue-contractor. With deny-by-default, no matching Allow policy means no access.

The Demo

config.toml Policy: only trusted-team can deploy
# Only agents owned by "user:trusted-team" can deploy. # All other agents hit deny-by-default. [[policy.policies]] id = "allow-trusted-deploy" effect = "allow" allowed_tools = ["deploy_service"] [policy.policies.principal_match] sub = "user:trusted-team" [policy.policies.intent_match] keywords = ["deploy"]
attack.sh rogue-contractor tries deploy_service
# Agent registered with owner: "user:rogue-contractor" # Session has deploy_service in authorized_tools curl -s -X POST http://127.0.0.1:8080 \ -H "Content-Type: application/json" \ -H "x-arbiter-session: $SESSION_ID" \ -H "Authorization: Bearer $TOKEN" \ -H "x-delegation-chain: user:rogue-contractor>$AGENT_ID" \ -d '{ "jsonrpc": "2.0", "id": 1, "method": "tools/call", "params": { "name": "deploy_service", "arguments": { "service": "payment-gateway", "environment": "production", "version": "9.9.9-backdoor" } } }'
response HTTP 403
{ "error": "POLICY_DENIED", "message": "No matching Allow policy for this tool call", "policy_trace": { "evaluated": 1, "matched": 0, "reason": "principal_match.sub: expected 'user:trusted-team', got 'user:rogue-contractor'" } }

The policy trace in the response shows exactly which policies were evaluated, why each failed, and what criteria would need to match. This transparency helps operators debug policy configurations without guessing.

The Defense

Arbiter stops this attack at Stage 8: policy evaluation in the arbiter-policy crate.

arbiter-policy -- Deny-by-Default Engine (Stage 8)

The policy engine evaluates all configured policies against the incoming request. Each policy can match on tool name, principal identity (via delegation chain), intent keywords, and parameter constraints. Policies are ranked by specificity score: more specific rules take priority over broader ones. If no Allow policy matches, the default action is deny. The engine returns a structured trace showing why each policy was skipped, which makes policy debugging deterministic rather than trial-and-error.

This is a critical distinction from traditional allow-by-default systems. In a zero-trust model, the administrator must explicitly grant access. Forgetting to write a policy results in no access, not unrestricted access. This is the correct failure mode for security infrastructure.

Preventing AI Agent
Resource Exhaustion

A runaway agent floods the upstream server with tool calls. Arbiter enforces per-session call budgets and per-minute rate limits, returning 429 after either limit is exceeded.

The Attack

AI agents can enter runaway loops. A prompt injection can instruct an agent to make thousands of API calls. A bug in the agent's reasoning loop can cause it to retry indefinitely. Without limits, one agent can exhaust the entire upstream service.

Resource exhaustion attacks do not require the agent to be malicious. A well-intentioned agent that encounters an error and retries aggressively can consume API quotas, rack up cloud costs, and degrade service for other agents and users. The upstream server has no way to distinguish a legitimate burst from an attack.

Traditional API rate limiting operates at the endpoint or IP level. AI agent resource exhaustion requires per-session limits: each agent session gets its own call budget and rate ceiling, independent of other sessions on the same IP or agent identity.

The Demo

Two scenarios: rate limit exhaustion (3 calls per minute) and total budget exhaustion (5 calls lifetime).

Part A: Rate limit 3 calls/minute, 4th call rejected
# Session created with rate_limit_per_minute: 3 # Calls 1-3 succeed, call 4 hits the rate limit Call 1: OK (passed rate check) Call 2: OK (passed rate check) Call 3: OK (passed rate check) Call 4: 429 RATE LIMITED
Part B: Budget exhaustion 5 total calls, 6th call rejected
# Session created with call_budget: 5 # Calls 1-5 succeed, call 6 exceeds the budget Call 1: OK (1/5 budget used) Call 2: OK (2/5 budget used) Call 3: OK (3/5 budget used) Call 4: OK (4/5 budget used) Call 5: OK (5/5 budget used) Call 6: 429 BUDGET EXHAUSTED
response HTTP 429
{ "error": "SESSION_INVALID", "message": "Session call budget exhausted" }

The Defense

Arbiter stops this attack at Stage 6: session enforcement, using two independent mechanisms.

arbiter-session -- Call Budget & Rate Limit (Stage 6)

Each session tracks two counters: a lifetime call count (compared against call_budget) and a sliding-window rate counter (compared against rate_limit_per_minute). Both checks are O(1) per request with no external dependencies. When either limit is exceeded, Arbiter returns HTTP 429 with error code SESSION_INVALID. The session is not destroyed -- it simply stops accepting new calls until the rate window resets or a new session is created.

Both limits are configured at session creation time, which means different agents can have different resource ceilings. A trusted internal agent might get a budget of 10,000 calls. An external contractor agent might get 50. The same proxy handles both.

session creation Configuring resource limits per session
{ "agent_id": "agent-xyz", "declared_intent": "read project files", "authorized_tools": ["read_file"], "call_budget": 5, // max 5 total calls "rate_limit_per_minute": 3, // max 3 calls per minute "time_limit_secs": 3600 }

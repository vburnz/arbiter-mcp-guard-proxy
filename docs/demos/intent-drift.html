Detecting AI Agent
Intent Drift in Real Time

An agent declares "read and analyze" as its intent. It passes session checks and policy evaluation. Then it calls write_file. Arbiter's behavioral anomaly detector catches the contradiction at Stage 9 -- the last line of defense.

The Attack

Intent drift is the most subtle attack pattern in this series. It exploits the gap between what an agent says it will do and what it actually does -- a gap that no other security control covers.

An agent declares a read-only intent ("read and analyze the source code") to obtain a session. The administrator, seeing a reasonable read-only intent, creates a session that includes both read and write tools -- perhaps because the agent's framework requires access to write_file for logging or scratch space. The policy engine has a broad Allow policy that covers all file operations.

Neither the session whitelist nor the policy engine catches the problem. The session allows write_file. The policy allows write_file. But the agent's declared intent was read-only. When it calls write_file to overwrite /etc/shadow, only the behavioral anomaly detector notices the mismatch.

This pattern is especially dangerous with prompt injection: an attacker injects instructions that cause the agent to gradually escalate from read operations to write operations, staying within the session and policy boundaries the entire time.

The Demo

First, a read_file call that matches the declared intent (passes all checks). Then, a write_file call that contradicts the intent.

setup Session with read-only intent but broad tool access
# Declared intent: "read and analyze the source code" # Authorized tools: [read_file, list_dir, write_file, delete_file] # Policy: allows all four tools # # Session check passes. Policy check passes. # Only behavioral analysis catches the contradiction.
legitimate call read_file matches "read and analyze" intent
"name": "read_file", "arguments": { "path": "/src/main.rs" } Result: PASSED (read_file matches "read" intent tier)
attack.sh write_file contradicts "read and analyze" intent
curl -s -X POST http://127.0.0.1:8080 \ -H "Content-Type: application/json" \ -H "x-arbiter-session: $SESSION_ID" \ -H "Authorization: Bearer $TOKEN" \ -d '{ "jsonrpc": "2.0", "id": 2, "method": "tools/call", "params": { "name": "write_file", "arguments": { "path": "/etc/shadow", "content": "root::0:0:root:/root:/bin/bash" } } }'
response HTTP 403
{ "error": "BEHAVIORAL_ANOMALY", "message": "Write operation in a Read session (intent: 'read and analyze the source code')" }

The Defense

Arbiter stops this attack at Stage 9 -- the last stage in the pipeline, after all other checks have passed. This is the behavioral anomaly detector in the arbiter-behavior crate.

arbiter-behavior -- Anomaly Detection (Stage 9)

The behavioral detector classifies the session's declared intent into operation tiers (read, write, admin) using keyword matching against configurable intent categories. The intent "read and analyze" matches the read tier. It then classifies each incoming tool call by operation type: read_file is a read operation, write_file is a write operation. When a write operation occurs in a session classified as read-only, the detector flags a behavioral anomaly. With escalate_anomalies = true, this flag becomes a hard deny: HTTP 403 with error code BEHAVIORAL_ANOMALY.

Defense in Depth

This demo illustrates why Arbiter uses 9 stages rather than a single authorization check. Session enforcement (Stage 6) verifies the tool is in the whitelist -- it was. Policy evaluation (Stage 8) checks for a matching Allow rule -- there was one. Only behavioral anomaly detection (Stage 9) catches the contradiction between the agent's declared intent and its actual operation. Each layer catches attacks that the previous layers cannot.

config.toml Behavioral anomaly detection with escalation
[sessions] escalate_anomalies = true # flag anomalies as hard deny # With escalate_anomalies = false, the anomaly would # be logged in the audit trail but the request would # proceed, allowing human review without blocking.

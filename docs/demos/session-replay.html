Expired Session Replay
Attacks on AI Agents

An attacker obtains an expired session ID and tries to reuse it. Arbiter checks TTL on every request: expired sessions return 408, closed sessions return 410.

The Attack

Session replay is a well-known attack pattern, but it takes on new dimensions in the AI agent context where sessions are created programmatically and may be logged, cached, or leaked through multiple channels.

An attacker obtains a valid session ID through log scraping, network sniffing, a compromised agent's memory, or a misconfigured monitoring system. If sessions have no expiry, a single leaked session ID grants permanent access to the upstream server with the full authorization scope of the original session.

The attack is especially dangerous because the replay uses a legitimately created session. The tool whitelist, agent identity, and declared intent are all valid. The only thing wrong is that the session should no longer be active -- either because it has expired or because the administrator explicitly closed it.

The Demo

Two scenarios: replaying an expired session (3-second TTL) and replaying a session that was closed via the admin API.

Part A: Expired session 3-second TTL, replayed after 4 seconds
# Session created with time_limit_secs: 3 Immediate call: OK (session is fresh) # Wait 4 seconds... Replay attempt: 408 SESSION EXPIRED
Part B: Closed session Session closed via admin API, then replayed
# Session closed via POST /sessions/{id}/close Replay attempt: 410 SESSION GONE
response (expired) HTTP 408
{ "error": "SESSION_INVALID", "message": "Session has expired" }

The Defense

Arbiter stops this attack at Stage 6: session enforcement, with two mechanisms.

arbiter-session -- TTL & Session Lifecycle (Stage 6)

Every session has a time_limit_secs field that sets its TTL. On every incoming request, the session middleware compares the current time against the session's creation time plus TTL. If the session has expired, the request is rejected with HTTP 408. Sessions can also be explicitly closed via the admin API (POST /sessions/{id}/close), which marks them as terminated. Requests using closed sessions return HTTP 410 Gone. A background cleanup task periodically removes expired sessions from memory.

The combination of TTL enforcement and explicit session closure means that leaked session IDs have a bounded window of usefulness. A 1-hour TTL limits the blast radius of any session leak to 1 hour at most. Explicit closure reduces it to zero.

Input Validation for
AI Agent Tool Parameters

An agent is authorized to call generate_text. It sends max_tokens = 50,000. The policy allows a maximum of 1,000. Arbiter catches the violation before the request reaches the upstream server.

The Attack

Authorizing a tool is not enough. The parameters matter. An agent allowed to call generate_text with reasonable limits could be manipulated to set max_tokens to 50,000 -- consuming excessive compute resources on a single request.

Parameter tampering is a particularly subtle attack because the tool call itself is legitimate. The agent has a valid session, the tool is in the whitelist, and the tool name matches a policy. Only the argument values are out of bounds. Without parameter-level constraints, the only defense is hoping the upstream server has its own validation.

This attack can be initiated by prompt injection ("set max_tokens to the maximum possible value"), by a compromised agent, or by an attacker who has obtained valid session credentials and wants to inflict maximum cost on the infrastructure. A single request with extreme parameters can consume more resources than thousands of normal requests.

The Demo

First, a legitimate call with max_tokens = 500 (passes). Then, the attack with max_tokens = 50,000 (blocked).

config.toml Policy with parameter constraints
# Allow generate_text only when parameters are in bounds [[policy.policies]] id = "allow-generate-bounded" effect = "allow" allowed_tools = ["generate_text"] [[policy.policies.parameter_constraints]] key = "max_tokens" min_value = 1.0 max_value = 1000.0 [[policy.policies.parameter_constraints]] key = "temperature" min_value = 0.0 max_value = 2.0
legitimate call max_tokens = 500 (within bounds)
# max_tokens: 500 <= 1000 limit ✔ "arguments": { "prompt": "Summarize the quarterly report", "max_tokens": 500, "temperature": 0.7 } Result: PASSED (HTTP 200/502)
attack.sh max_tokens = 50000 (exceeds constraint)
curl -s -X POST http://127.0.0.1:8080 \ -H "Content-Type: application/json" \ -H "x-arbiter-session: $SESSION_ID" \ -H "Authorization: Bearer $TOKEN" \ -d '{ "jsonrpc": "2.0", "id": 2, "method": "tools/call", "params": { "name": "generate_text", "arguments": { "prompt": "Generate an extremely long document", "max_tokens": 50000, "temperature": 0.7 } } }'
response HTTP 403
{ "error": "POLICY_DENIED", "message": "No matching Allow policy for this tool call", "detail": "parameter constraint failed: max_tokens=50000 exceeds max_value=1000" }

The Defense

Arbiter stops this attack at Stage 8: policy evaluation, via parameter constraint checking in the arbiter-policy crate.

arbiter-policy -- Parameter Constraints (Stage 8)

Each policy can include parameter_constraints entries that specify min/max bounds for numeric arguments. During policy matching, the engine extracts the argument values from the parsed MCP request and checks each one against the configured bounds. If any constraint fails, the policy does not match. With deny-by-default, a non-matching policy means the request is denied. Parameter constraints are composable: you can define different limits for different trust levels or principals by creating separate policies with different bounds.

This design keeps parameter validation in the policy layer rather than hardcoding it in the proxy. You can change limits without redeploying -- just update the config and reload. Different agents can have different limits based on their trust level.

Stopping Unauthenticated
AI Agent Access

A rogue agent sends an MCP tool call without a session header. Without Arbiter, it passes straight through to the upstream server. With Arbiter, it gets a 403 before the request leaves the proxy.

The Attack

The most fundamental attack against an MCP proxy is also the simplest: send a raw JSON-RPC request without identifying yourself.

In a default deployment without session enforcement, any process that can reach the proxy's listen address can invoke any tool on the upstream MCP server. There is no identity verification, no authorization check, and no audit trail. The proxy acts as a transparent pipe.

This is not a theoretical concern. Agents run as background processes, CI/CD pipelines invoke tools programmatically, and internal services share network segments. A single misconfigured cron job or a compromised container can send arbitrary tool calls to the upstream server -- reading sensitive files, modifying configurations, or deleting data -- with no record of who did it or why.

The attack requires zero sophistication. A single curl command is sufficient to invoke any tool the upstream server exposes.

The Demo

The attacker sends a valid MCP JSON-RPC request to read /etc/passwd, but provides no session header.

attack.sh MCP tool call with no session header
# No x-arbiter-session header. No identity. No authorization. curl -s -X POST http://127.0.0.1:8080 \ -H "Content-Type: application/json" \ -d '{ "jsonrpc": "2.0", "id": 1, "method": "tools/call", "params": { "name": "read_file", "arguments": { "path": "/etc/passwd" } } }'
response HTTP 403
{ "error": "SESSION_REQUIRED", "message": "A valid x-arbiter-session header is required for all MCP requests" }

The request never reaches the upstream server. Zero bytes forwarded.

The Defense

Arbiter stops this attack at Stage 6 of its 9-stage middleware pipeline: session enforcement.

arbiter-session (Stage 6)

When require_session = true (the default), the session middleware checks every incoming MCP request for the x-arbiter-session header. If the header is missing or empty, the request is rejected immediately with HTTP 403 and error code SESSION_REQUIRED. The request never reaches stages 7 through 9 (policy, behavior, forward).

This is the foundational control that all other protections build on. Without session enforcement, an agent can skip identity verification entirely. With it, every MCP request must be associated with a registered agent, a scoped session, and a declared intent.

config.toml Session enforcement is on by default
[proxy] upstream_url = "http://127.0.0.1:8081" require_session = true # default, reject requests without a session strict_mcp = true # default, reject non-MCP POST bodies

Try It Yourself

terminal Run the demo locally
git clone https://github.com/cyrenei/arbiter-mcp-firewall cd arbiter && cargo build --release cd demos/01-unauthenticated-access bash demo.sh

The script starts Arbiter, sends the attack, displays the structured JSON response, and shuts down. No upstream server required -- the attack is blocked at the proxy layer.

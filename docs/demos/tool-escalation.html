AI Agent Tool Escalation:
Attack and Defense

An agent authorized for read_file and list_dir attempts to call delete_file. Arbiter's per-session tool whitelist stops the escalation before the request leaves the proxy.

The Attack

Tool escalation is the privilege escalation of the AI agent world. An agent starts with a narrow set of authorized tools and attempts to invoke tools outside that set.

In this scenario, an administrator creates a session authorizing an agent to use only read_file and list_dir. The agent first makes a legitimate read_file call, which succeeds. Then it attempts to call delete_file -- a destructive operation outside its authorized scope.

Without session-scoped tool whitelisting, a compromised or misbehaving agent can invoke any tool the upstream server exposes. This is especially dangerous when agents have access to broad tool catalogs that include both read and write operations. A single prompt injection can redirect the agent from reading data to deleting it.

The Demo

First, a legitimate read_file call (which passes). Then, the attack: delete_file.

setup Session scoped to read-only tools
# Create session with explicit tool whitelist curl -s -X POST http://127.0.0.1:3000/sessions \ -H "x-api-key: demo-key" \ -H "Content-Type: application/json" \ -d '{ "agent_id": "agent-xyz", "declared_intent": "read and list project files", "authorized_tools": ["read_file", "list_dir"], "time_limit_secs": 3600, "call_budget": 100 }'
attack.sh Attempt to call delete_file (not in whitelist)
curl -s -X POST http://127.0.0.1:8080 \ -H "Content-Type: application/json" \ -H "x-arbiter-session: $SESSION_ID" \ -H "Authorization: Bearer $TOKEN" \ -d '{ "jsonrpc": "2.0", "id": 2, "method": "tools/call", "params": { "name": "delete_file", "arguments": { "path": "/etc/passwd" } } }'
response HTTP 403
{ "error": "SESSION_INVALID", "message": "Tool 'delete_file' is not in this session's authorized tool set" }

The read_file call passed (HTTP 200/502). The delete_file call was blocked (HTTP 403). The session remains valid for future authorized calls.

The Defense

Arbiter stops this attack at Stage 6: session enforcement in the arbiter-session crate.

arbiter-session -- Tool Whitelisting (Stage 6)

Every session is created with an explicit authorized_tools list. When a tool call arrives, the session middleware extracts the tool name from the parsed MCP request and checks it against this whitelist. If the tool is not in the set, the request is rejected with SESSION_INVALID. The check is O(1) -- it uses a HashSet lookup -- and the session is not invalidated, so the agent can still make calls with authorized tools.

This implements the principle of least privilege at the tool-call level. Even if an agent is authenticated, has a valid session, and passes policy evaluation, it can only invoke the specific tools it was granted when the session was created.
